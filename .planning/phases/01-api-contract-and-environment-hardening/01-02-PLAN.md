Allowed tools/actions:
- Use read-before-write, evidence-before-action, minimal diff, verify-before-report.
- Refactor inside `src/index.ts` only when it directly supports validation, env typing, or response safety.
- Run `npm run typecheck` and the frontmatter verification commands.

Forbidden actions:
- Do not change route paths or HTTP methods.
- Do not add dependencies.
- Do not put secrets in logs or client responses.
- Do not weaken TypeScript strictness or add unnecessary `as any` escape hatches.
- Do not edit config/docs; Plan 01-01 and Plan 01-03 own those files.

Implementation sequence:
1. Read Plan 01-01 summary and confirm canonical names: `CF_API_TOKEN`, `PHOENIX_DEBUG_TOKEN`, `PHOENIX_DEBUG_ENABLED`, `MODEL_PROVIDER=workersai|azureopenai`.
2. Update `Env` so secret fields are optional where appropriate and include `PHOENIX_DEBUG_ENABLED?: string` and `PHOENIX_DEBUG_TOKEN?: string`.
3. Replace broad request body trust with validation helpers:
   - `MAX_MESSAGE_LENGTH = 4000`
   - `MAX_HISTORY_MESSAGES = 12`
   - `MAX_HISTORY_MESSAGE_LENGTH = 2000`
   - `MAX_USER_PROFILE_JSON_LENGTH = 4000`
   - body must be a plain object, not null or array
   - `message` must be a non-empty string after trim and no longer than the max
   - `history` is optional, must be an array if present, max 12 entries, each role exactly `user` or `assistant`, each content non-empty string and no longer than the max
   - `userProfile` is optional, must be a plain object if present, and its JSON string length must be <= 4000
   - `debug` is optional boolean; non-boolean debug values are rejected
4. Add trusted debug authorization:
   - trusted debug is true only when `PHOENIX_DEBUG_ENABLED` is `"true"` and request header `X-Phoenix-Debug-Token` exactly matches non-empty `env.PHOENIX_DEBUG_TOKEN`
   - if `debug: true` is requested without trust, return `403` JSON with a clear generic message before retrieval/model calls
   - successful debug payload may include provider/model, chunk count or source metadata, and retrieval error summary, but must not include raw `systemPrompt`
5. Keep `GET /api/health`, `POST /api/coach`, and CORS preflight behavior intact.
6. Preserve provider routing for `workersai` and `azureopenai`, using `cf-aig-gateway-id` for Workers AI. Unsupported provider values still return 400.
7. Sanitize upstream provider failure responses so raw provider body text is logged server-side but not returned to clients. If trusted debug is active, expose only bounded diagnostic metadata, not secrets or prompts.
8. Run verification.

Required interfaces/content structure:
- Add a small `jsonResponse(env, status, payload)` helper only if it reduces duplication without changing response shape.
- Return validation failures as JSON with `{ error: "Bad Request", message: string }`.
- Preserve successful coach response shape as `{ response: string }`, with optional trusted `{ debug: ... }`.
- Preserve health response as `{ ok: true, service: "phoenix-ai-api" }`.

Edge/error cases:
- Invalid JSON returns 400.
- JSON arrays, null, strings, and numbers return 400.
- Whitespace-only `message` returns 400.
- Overlong `message` returns 400.
- Excessive history returns 400.
- `history` role `system` returns 400.
- Unauthorized debug returns 403 before retrieval/model calls.
- Missing provider credentials return configuration errors without printing secret values.

Verification criteria:
- `npm run typecheck` passes.
- Static checks confirm validation/debug tokens exist and raw `systemPrompt` response output is removed.
- Static checks confirm route literals remain present.

Final result format:
- Status: completed or BLOCKED.
- Files changed.
- Verification commands and results.
- Any decisions or issues.
